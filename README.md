# Co-build

Welcome to **Co-build** — a live collaborative building space for the Cursor event.

This repository is open to all participants who want to contribute ideas, code, documentation, prompts, experiments, or demos that make the event more useful, more creative, and more fun.

During the event, we build together with Cursor, submit contributions through Pull Requests, and select some of them to be reviewed, merged, and even run live on screen like a real-time build match.

---

## What is Co-build?

Co-build is a shared GitHub repository created for the event so participants can collaborate in public during the session.

You can contribute:

- Ideas
- Features
- UI components
- Small tools
- Prompts for Cursor
- Documentation
- Sample apps
- Demos
- Fixes and improvements
- Anything else that benefits the event

The goal is simple:

**build together, learn together, and ship together — live.**

---

## How it works

1. Join the repository
2. Pick an idea or propose your own
3. Build with Cursor during the event
4. Submit your contribution via Pull Request
5. Maintainers review selected submissions
6. Some contributions may be merged and showcased live on screen

This makes the event feel like an open-source sprint mixed with a live builder match.

---

## What kinds of contributions are welcome?

We welcome contributions that are:

- Useful for event participants
- Interesting to build live
- Easy to understand and demo
- Small enough to ship quickly during the event
- Creative, practical, or inspiring

Examples:

- A helpful event dashboard
- A mini tool for participants
- A live voting widget
- A prompt library for Cursor
- A simple interactive demo
- Better docs for attendees
- A showcase page for live projects
- Utilities that help people collaborate faster

---

## Contribution rules

Please keep contributions:

- Relevant to the event
- Respectful and collaborative
- Small or modular enough to review quickly
- Clear in purpose
- Safe to run and demo

Avoid:

- Breaking the main project
- Large unreviewable changes
- Spam or joke submissions that block others
- Harmful or unsafe code
- Content unrelated to the event

If you are not sure, open an issue or submit a draft Pull Request first.

---

## How to contribute

### 1. Fork the repository
Create your own fork of this repo.

### 2. Create a branch
Use a clear branch name, for example:

- `feature/live-voting-widget`
- `docs/cursor-prompt-guide`
- `idea/project-showcase-page`

### 3. Build your contribution
Use Cursor and your preferred workflow to create something valuable for the event.

### 4. Submit a Pull Request
Open a Pull Request with:

- A clear title
- A short description of what you built
- Why it helps the event
- Screenshots or demo notes if relevant

### 5. Review and live selection
Maintainers may review, comment, merge, or nominate your contribution for a live showcase.

---

## Pull Request template

When opening a Pull Request, please include:

### Summary
What did you build or improve?

### Why it matters
How does this help the event or participants?

### Type of contribution
- Idea
- Code
- Docs
- Prompt
- Demo
- Fix
- Other

### Demo notes
How should we test or present it live?

---

## Live showcase criteria

Some Pull Requests may be selected to run or be presented live during the event.

We will prioritize contributions that are:

- Working or easy to demonstrate
- Visually clear
- Useful or surprising
- Built fast and creatively
- Good examples of collaborative building with Cursor

This is not only about polished code.
It is also about speed, creativity, usefulness, and collaboration.

## Suggested folders

You may organize contributions into folders like:

- `/ideas` — concept notes and proposals
- `/docs` — guides, instructions, and event documentation
- `/prompts` — useful Cursor prompts and workflows
- `/apps` — demos, mini tools, and prototypes
- `/components` — reusable UI or feature components
- `/experiments` — fast builds and creative tests
